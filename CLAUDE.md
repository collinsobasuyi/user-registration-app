## Project Purpose

A full-stack user registration application built as a learning project. The goal is to demonstrate how a real web application connects a frontend, backend API, database, version control, and deployment platform.

**The chain:**
```
Browser (Next.js) → HTTP → FastAPI (Python) → SQLAlchemy → PostgreSQL (Neon)
```

## Database Schema

```sql
CREATE TABLE users (
  id               SERIAL PRIMARY KEY,
  full_name        VARCHAR NOT NULL,
  email            VARCHAR UNIQUE NOT NULL,
  hashed_password  VARCHAR NOT NULL,
  created_at       TIMESTAMPTZ DEFAULT NOW()
);
```

SQLAlchemy creates this table automatically on startup via `models.Base.metadata.create_all(bind=engine)`. No manual SQL needed.

Passwords are hashed with bcrypt before storage. The plain-text password is never saved.

## API Endpoints

Base URL (local): `http://localhost:8000`

| Method | Endpoint | Description | Status |
|---|---|---|---|
| GET | `/` | Health check | 200 |
| GET | `/api/users` | List all users | 200 |
| GET | `/api/users/{id}` | Get user by ID | 200 / 404 |
| POST | `/api/register` | Create new user | 201 / 400 |
| PUT | `/api/users/{id}` | Full update (name + email) | 200 / 400 / 404 |
| PATCH | `/api/users/{id}` | Partial update (name or email) | 200 / 400 / 404 |
| DELETE | `/api/users/{id}` | Delete user | 204 / 404 |

Interactive docs are available at `http://localhost:8000/docs` (Swagger UI, auto-generated by FastAPI).

### POST /api/register

```json
{
  "full_name": "Jane Smith",
  "email": "jane@example.com",
  "password": "password123"
}
```

### PUT /api/users/{id}

All fields required:
```json
{
  "full_name": "Jane Updated",
  "email": "jane.new@example.com"
}
```

### PATCH /api/users/{id}

Send only the fields you want to change:
```json
{
  "full_name": "Jane Only Name Changed"
}
```

## Key Design Decisions

**Why bcrypt directly instead of passlib?**
passlib 1.7.4 uses `bcrypt.__about__.__version__` which was removed in bcrypt 4.0. Rather than pinning to an old bcrypt version, bcrypt is used directly — it is simpler and has no compatibility issues.

**Why Neon instead of local PostgreSQL?**
Neon provides a free serverless PostgreSQL instance with zero local setup. The connection string includes `?sslmode=require` which is passed through to psycopg2. The SQLAlchemy engine uses `pool_pre_ping=True` to handle Neon's idle connection drops.

**Why not use Next.js API routes for the backend?**
The explicit goal of this project is to learn how a separate frontend and backend communicate over HTTP. Keeping them separate teaches request/response, CORS, environment variables, and independent deployment.

**CORS configuration**
The backend allows `http://localhost:3000` and `http://127.0.0.1:3000` in development. For production, this must be updated to the deployed Vercel URL.

## Running Locally

### Backend

```bash
cd backend
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
# Add your Neon DATABASE_URL to .env
./venv/bin/uvicorn main:app --reload
```

Runs at `http://localhost:8000`

### Frontend

```bash
cd frontend
npm install
cp .env.local.example .env.local
# NEXT_PUBLIC_API_URL=http://localhost:8000 is already set
npm run dev
```

Runs at `http://localhost:3000`

## What This Project Teaches

- How a browser talks to a backend API over HTTP
- How FastAPI validates requests using Pydantic schemas
- How SQLAlchemy maps Python classes to PostgreSQL tables
- Why passwords must be hashed and never stored as plain text
- How environment variables keep credentials out of source code
- The difference between PUT (replace all) and PATCH (update some)
- How CORS works and why it exists
- How to deploy a full-stack app across separate platforms

## Known Issues / Limitations

- No authentication. Any request can read, update, or delete any user. A real app would require a login flow and JWT tokens before exposing these endpoints.
- No pagination on `GET /api/users`. Fine for learning, would need limits for production.
- Password cannot be updated via PUT or PATCH. A separate change-password endpoint with current password verification would be needed.
